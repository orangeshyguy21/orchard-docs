## Dashboard

Get an executive overview of your mint's health and status:

- **Info** — the mint's public info
- **Config** — supported features
- **Keysets** — keyset breakdown and database growth.
- **Activity** — recent activity and volume.
- **Balance Sheet** — a real-time snapshot of the mint's holdings.
- **Nutalytics** — rich timeseries charts of the mint's activity.

## Info

Update public information that wallets read about your mint.

- **Name**, **description**, and a longer **description**.
- **Icon** — the image a wallet displays for your mint.
- **Connection URLs** — where wallets reach the mint.
- **Contacts** — how users get in touch (email, X, Nostr, and so on).
- **Message of the Day** — a short broadcast note.

Keep it accurate: this is what a wallet shows someone before they trust your mint.

## Config

View and manage the features your mint does and does not support, and how they are configured.
Some features are managed in the mint's own config file, and others can be managed in Orchard.

<Aside type="note" title="Why the split">
The mint's RPC offers only limited real-time control over mint features for now, so anything
it doesn't expose has to be set in the config file directly.
</Aside>

## Keysets

Manage your mints keysets and monitor their activity over time as you rotate them in and out.
Keysets can be rotated to change fees, and to manage the growth of the mint's database.

## Database

Back up and restore the mint database straight from the dashboard, so keeping a current copy
doesn't mean dropping to the command line:

- **Create Backup** downloads a copy of the database.
- **Restore Backup** loads one back in — shut the mint down first, match the backup to the
  mint's version, restore, then start it again.

The same screen also browses the underlying mint, melt, and swap records, filterable by
date, unit, and state.

### The recovery model

A mint is restored from two things together:

- **The seed phrase**, which derives the mint's signing keys. It never changes, lives in the
  mint daemon's config rather than its database, and Orchard never holds it — so this half
  stays your responsibility (see the mint's [seed step](/new-mint/mint/#generate-the-mint-seed)).
- **The database**, which changes every time the mint issues or redeems ecash — the half
  Orchard backs up for you.

With both, a mint lost to a dead disk comes back fully operational, with the same keysets and
the same outstanding ecash. With only one, it doesn't. Take backups on a schedule and store
them off the machine, separate from your seed backup.
